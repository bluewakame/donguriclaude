# セキュリティ評価レポート（2026-03-10）

## 対象
- Next.js + NextAuth + Prisma を利用した `donguri-app`
- 主に API ルート、認証・認可、トークン残高更新ロジックを確認

## 総評
- 認証・認可の基本実装（`auth()` によるログイン確認、管理者・店舗オーナー権限チェック）は実装されている。
- 一方で、**残高更新まわりの競合対策不足**、**CSRF/レート制限対策不足**、**依存関係更新リスク** が目立つ。

## 主要な指摘事項

### 1) 競合状態による二重消費/不整合リスク（High）
- `exchangeLeaves` は残高確認をトランザクション外で実施し、その後で減算している。
- 同時リクエスト時に「両方が事前チェックを通過」して過剰減算を引き起こす可能性がある。
- `shop/spend` でも同様に残高確認と更新が分離しており、二重購入リスクがある。

**推奨対策**
- 残高条件付き更新（例: `updateMany where id=? and balance >= cost`）を使い、更新件数で成否判定。
- DB 制約（`CHECK (balance >= 0)`）を追加し、アプリバグ時も負残高を防止。

### 2) レート制限未実装（High）
- ログイン・登録・QRスキャン・購入など主要 API にレート制限がない。
- 資格情報の総当たり、QRトークン推測試行、アプリ資源の消耗攻撃に弱い。

**推奨対策**
- IP + ユーザーID ベースで API ごとに制限（例: 1分/5分単位）。
- 失敗回数増加時にクールダウン・Captcha を導入。

### 3) CSRF 防御の明示不足（Medium）
- 複数の状態変更 API が Cookie セッション前提で実行されるが、`Origin/Referer` の検証や独自 CSRF トークン検証が確認できない。
- `SameSite` だけに依存すると、将来の設定変更やブラウザ差異で事故が起こりうる。

**推奨対策**
- 状態変更 API に対し `Origin` チェックを共通ミドルウェア化。
- 重要操作に二重送信トークン（double-submit）等を導入。

### 4) 依存関係のセキュリティ更新遅延リスク（Medium）
- `package-lock.json` 内に、特定バージョンの脆弱性注意文が含まれる。
- `next` が固定で `14.2.5` のため、上流のセキュリティパッチ追従が必要。

**推奨対策**
- Next.js / next-auth / prisma を定期更新する運用（最低月1回）を設定。
- CI に `npm audit` または SCA ツールを組み込み、失敗閾値を定義。

## 良い実装
- 管理者 API は `ADMIN_EMAILS` で明示的に権限判定をしている。
- 店舗 QR 更新 API は `createdBy` 一致を検証しており、他店舗操作を防いでいる。
- Cron API は本番で `CRON_SECRET` による Bearer 検証を行っている。

## 優先対応順
1. 残高更新の競合対策（条件付き更新 + DB CHECK）
2. ログイン/スキャン/購入系へのレート制限
3. CSRF 防御の共通化
4. 依存関係アップデート運用の定着
