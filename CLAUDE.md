# プロジェクトの意思決定（donguri-app）

## なぜこの構成なのか
- 学習・PoC段階でも実運用を想定し、Next.js App Router + API Route で単一リポジトリ内にUIとAPIを同居させている。
- 認証（NextAuth）とDB（Prisma）を先に統合する方針を取り、機能追加時の認可漏れを防ぎやすくしている。

## 絶対にやってはいけないこと
- どんぐり残高更新ロジックは競合が発生しやすいため、残高確認と更新を分離した実装に戻さない。
- 認可（admin/merchant）の判定をUI側だけで完結させない。必ずAPI側で再検証する。
- 本番環境で `CRON_SECRET` を未設定のままcron APIを有効化しない。

## 運用とデプロイ
- 本番はVercel前提。環境変数はVercel上に設定し、`.env*` をコミットしない。
- DBマイグレーションは必ず開発/検証環境で適用確認後に本番へ進める。

## チームワークフロー
- 変更は必ず最小差分で行い、関連するlint/testコマンドを実行してからコミットする。
- セキュリティ関連変更（認証・決済相当・残高更新）はPR説明にリスクとロールバック方針を記載する。

## ビジネスコンテキスト
- 「来店体験を阻害しない」ことが優先。チェックイン・QRスキャンの失敗率上昇につながる重い処理は避ける。
- トークン/残高不整合は信頼失墜に直結するため、UX改善より整合性を優先する。
